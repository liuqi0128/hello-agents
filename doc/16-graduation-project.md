# 第十六章 毕业设计：构建你的 Node.js 多 Agent 应用

对应原教程：`docs/chapter16/第十六章 毕业设计.md`。毕业设计的目标不是堆叠最多框架，而是交付一个有明确用户、可演示工作流、可验证质量与清晰边界的 Agent 应用。

## 16.1 选题标准

选题应同时满足：真实痛点、可获得的合法数据/工具、两到四周内可完成的最小闭环、可量化的成功标准和可说明的安全边界。

可选方向：代码审查助手、个人知识库问答、会议行动项 Agent、旅行规划助手、运营报表分析、客服草稿助手、研究报告 Agent。优先选你能获得真实任务样本的方向。

## 16.2 推荐项目结构

```text
my-agent-project/
  src/
    agents/          # 协调与角色策略
    tools/           # 经验证的外部能力
    schemas/         # Zod 输入/输出契约
    services/        # 业务与持久化
    api/             # HTTP/SSE/WebSocket 路由
  tests/             # 单元、集成与评测任务集
  evals/             # JSONL 保留集和评分器
  docs/              # 架构、威胁模型、演示说明
  .env.example
  package.json
  README.md
```

初始化：

```bash
npm init -y
npm pkg set type=module
npm install openai dotenv zod express
npm install -D vitest eslint
```

`.env.example` 只写变量名，例如 `LLM_API_KEY=`、`LLM_MODEL=`、`DATABASE_URL=`；绝不提交真实密钥。

## 16.3 最小交付清单

- README：问题、目标用户、架构图、安装、配置、运行、示例与限制。
- 一个端到端用户流程：输入 -> Agent/工具 -> 结构化结果或可审计报告。
- 至少一个受控工具，包含输入校验、超时和错误处理。
- 至少十条评测任务，覆盖正常、边界、失败和安全输入。
- 自动化测试、日志字段说明、数据与密钥处理方式。
- 3 至 5 分钟演示：说明成功案例，也展示一个系统如何安全失败。

## 16.4 评分与迭代

用第十二章的评测器为项目定义基线。每次改变 Prompt、模型、工具或上下文策略后，在同一保留集上重新运行；记录成功率、成本、延迟和高风险失败。不要根据演示中单次成功就宣布功能完成。

建议的验收矩阵：

| 维度 | 可验证问题 |
| --- | --- |
| 正确性 | 结果是否满足任务评分器或业务规则？ |
| 可靠性 | 超时、无效工具参数和外部失败是否被处理？ |
| 安全性 | 用户能否诱导越权、访问其他用户数据或触发副作用？ |
| 可用性 | 用户是否知道结果依据、下一步和需要确认的操作？ |
| 可维护性 | 模型、Prompt、工具和任务集是否可版本化与回归测试？ |

## 16.5 提交前检查

1. `npm test` 通过，评测报告已保存且写明环境版本。
2. `.env`、日志、截图和演示数据不含密钥或个人信息。
3. README 从空环境按步骤可复现；每个依赖都有用途。
4. 检查工具权限、网络访问、文件路径和写操作确认点。
5. 使用小而聚焦的提交记录；提交 PR 时说明功能、测试、风险与演示方式。

## 16.6 示例：代码审查 Agent 的边界

一个合格的代码审查 Agent 可读取限定仓库中的 diff，运行只读静态分析，将“发现”按严重性、证据和建议输出。它不应自行推送代码、修改 CI、读取工作区外文件或将私有代码发给未批准的模型服务。

完成项目后，将重点放在“哪一项任务改善了、哪一项仍不可靠、为何如此”。这种可验证的工程说明比更长的提示词更能体现 Agent 开发能力。
